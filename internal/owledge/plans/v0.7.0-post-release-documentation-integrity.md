# v0.7.0 Post-Release Documentation Integrity

## Goal

Make the public source-of-truth documentation unambiguously describe Owledge
v0.7.0 and prevent a future release from publishing a package while the
README, version badge, or release documentation remain stale.

## Scope

- Audit the root version sources, README badge, active documentation, release
  workflow, package metadata, tag, and remote branch state.
- Preserve clearly historical release plans, changelog entries, archive files,
  fixtures, and component-local version fields.
- Strengthen the release gate only where it cannot already detect the reported
  failure mode.

## Phases and QA gates

### Phase 1 — Establish release truth

- [ ] Compare `VERSION`, `pyproject.toml`, README badge, changelog, tag, and
  remote `main`.
- [ ] Classify every older active-doc version mention as historical or stale.

Gate: `python tools/owledge.py test release-trust --project-root .`

### Phase 2 — Correct and harden

- [ ] Correct public stale claims, if present.
- [ ] Add a regression check for any gap found in the current gate.

Gate: `python tools/owledge.py test public-docs --project-root .` and
`python tools/owledge.py test release-trust --project-root .`

### Phase 3 — Publish verification

- [ ] Verify the final commit is on `main`, aligned with the v0.7.0 tag, and
  ready to push.
- [ ] Record final evidence in the workpackage.

Gate: `python tools/owledge.py test publish-readiness --project-root .`

## Definition of Done

The tracked public README and active docs use v0.7.0 where they state the
current release; historical references remain explicitly historical; version
gates pass; and the final commit is published on `main`.

## Resume State

Current phase: Phase 1.
