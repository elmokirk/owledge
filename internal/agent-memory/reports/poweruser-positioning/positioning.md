# Poweruser Positioning Scorecard

Scores describe fit for Owledge's memory/proof positioning, not absolute product quality.

| Comparator | Category | Score | Positioning | Limitation |
| --- | --- | ---: | --- | --- |
| Owledge | Markdown-first memory/proof layer | 90 | Best when long-running projects need durable plans, evidence, reviews, handoffs, decisions, and proof artifacts across agents. | Not an execution runtime, graph database, vector store, or replacement for existing project-specific agent instructions. |
| Superpowers-style execution frameworks | Agent execution methodology | 60 | Strong for coding workflow discipline, implementation flow, and agent execution structure. | Usually not a project-wide reviewed memory contract or cross-project proof layer. |
| Graphify-style graph visualizers | Graph visualization and exploration | 55 | Strong for seeing relationships, dependencies, and clusters once clean source records exist. | A graph view should consume reviewed memory; it should not become the canonical truth by itself. |
| LLM Wiki / Obsidian-style knowledgebases | Human-curated notes and wiki structure | 60 | Strong for human knowledge capture, linking, and existing taxonomy. | Needs explicit contracts, review status, and scoped context generation before agents can consume it reliably. |
| Claude Code / Codex-style agent runtimes | Agent runtime and coding execution | 68 | Strong for executing tasks, editing files, running tests, and operating agent sessions. | Runtime memory and logs are not the same as reviewed project memory with reusable decisions and lessons. |
| Custom harnesses and hooks | Team-specific agent automation | 52 | Strong for highly tailored orchestration and local automation. | Often depends on implicit team conventions unless paired with a stable memory and evidence contract. |

## Interpretation

Owledge is strongest as a durable memory, evidence, context hygiene, and traceability layer beside execution runtimes, graph views, wikis, and harnesses.

## Boundary

This is a snapshot-first local scorecard. It should be updated when concrete comparator repositories, versions, or benchmark evidence are selected.
