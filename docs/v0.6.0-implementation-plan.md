### 1.1 Project modes + abstract + planning_mode (FB-005 + FB-017, merged)

FB-005 and FB-017 are the same concept. Merge into one implementation.

**Files:**
- `PROJECT_CONTEXT.template.md` — add 3 frontmatter fields after `project_id`:
  - `project_mode: "poc"` (enum: poc, mvp, side, saas; default poc)
  - `project_abstract: ""` (string, 1-3 sentences)
  - `planning_mode: "supervised"` (enum: supervised, approve-automatically, full-access; default supervised)
- `PROJECT_CONTEXT.template.md` — add a `## Project Mode` section after `## Project Vision` documenting the 4 modes with intuitive definitions (from FB-017 action):
  - **poc** — prove it works; minimal plan, no cutline, abstract optional, no metrics
  - **mvp** — one user gets value; full cutline, evidence, acceptance criteria, ≥1 metric
  - **side** — fun and bounded; lightweight goals/non-goals, abstract recommended, metrics optional
  - **saas** — ship and charge; full abstract, ≥3 metrics with decision rules, compliance notes
- `PROJECT_CONTEXT.template.md` — add a `## Planning Mode` section documenting the 3 permission modes (from FB-009)
- `docs/mvp-plan-example.md` — add `## Project Abstract` placeholder after `# Example MVP Plan`

**Acceptance (FB-005 + FB-017):** A new project with `project_mode: saas` forces the agent to draft an abstract and success metrics before any plan is accepted. A user reading the template understands which mode to pick without external docs.

**Test:** `python tools/agent_memory_cli.py --project-root . validate-memory --strict` passes with the new fields; `owledge init-project --target <temp>` produces a `PROJECT_CONTEXT.md` with the 3 new fields.
